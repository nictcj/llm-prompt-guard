# LLM Prompt Guard
This demo project showcases a lightweight testing framework designed to evaluate a mock LLM’s behaviour. Its main focus is on test execution flow, reporting, conversation logging, test inventory management, and CI pipeline integration. Only a small set of test cases has been created intentionally at this stage, to verify that the framework, tooling, and CI workflow operate correctly. Test coverage will be expanded incrementally as the framework matures.

Because this is a mocked demo project, the business logic behind the placeholder agents — MockLLM, Prompter, and Evaluator — is currently hardcoded to simulate representative behaviours. They should not be mistaken for fully developed agentic components. A natural next step would be to move this logic into structured data tables for better maintainability and test case reuse, with the option of eventually integrating an actual open-source LLM.

Components summary as below:
- [Mock-LLM](#mock-llm) that accepts user prompt and returns responses.
- [Prompter](#prompter) component to generate agentic request prompts.
- [Evaluator](#evaluator) component to judge whether a prompt contains sensitive information.
- [FastAPI](#fast-api) backend service layer for API access.
- [Webpage](#web-ui) that allows user to enter prompt and send API requests to endpoint and receive responses.
- [Playwright/PyTest](#harness) testing framework to design and run autotests on browser.
- [Reporting](#reporter) function that can generate:
    - A complete source-of-truth catalogue report for all test cases for test manager overview.
    - Test execution report after each test run.
- [Conversation Runner](#conversation-runner) helper that logs the conversation between user and agent.
- [CI Pipeline](#ci) to autotrigger CI tests following each push/pull requests.

### Conversation Runner/Logger
To simulate a real conversation with a real LLM agent (ChatGPT) and to test its secret-guarding reliability, a set of instructions was given to the agent at the beginning to protect the user-defined secret. Following that, another sequence of prompt injection attacks was then carried out to try and trick the agent in exposing the secret without the usage of the required passcode for revelation. The reports below were generated by plugging in the Conversation Runner helper into a pytest with prompts and responses that mirrored the actual agent conversation.

#### Conversation #1
The agent exposed the secret rather easily at the mention of a phrase that resembled the passcode. It even explicitly highlighted the exact word that resembled the passcode.\

#### Conversation #2
The same set of instructions was given to the agent again. This time around the agent seems to retained certain context from the previous conversation and changed its behaviour to better guard the secret. A mention of the similar passcode phrase no longer gave out explicit hint of it. More malicious prompts were then thrown at the agent and ultimately the secret was still revealed after the initial passcode was effortlessly overwritten with a new one.\

## <a id="quickstart"></a> Quickstart
This project was developed on Python 3.13, significantly older versions may not be compatible.\
To install all the required dependencies at-once:

Run
```
pip install -r requirements.txt
```
and
```
playwright install --with-deps
```

You will need both the API backend and webpage to be running to run the tests:
```
uvicorn main:app --reload
```
```
python -m http.server 5500
```

To run tests:
```
pytest -v
```

## <a id="mock-llm"></a> Mock-LLM (Agent) [↑](#top)
This layer is represented by a MockLLM class object that accepts information from the user, through the web UI, and then returns responses based on internally defined rules. Each MockLLM instance can contain a user-defined secret keyword that is supposed to be classified as sensitive information and should never be leaked through responses.

## <a id="prompter"></a> Prompter (Agent) [↑](#top)
This component mimics an agent that would produce prompt sentences with input keywords. In an ideal scenario this would be another independent LLM agent that can interpret user intention and produce concise prompts to be fed to another LLM agent for responses. In this environment, however, the prompts will be pre-defined and controlled for the sake of simplicity.

## <a id="evaluator"></a> Evaluator (Agent) [↑](#top)
Much like the Prompter agent, this component would process the response returned by the MockLLM after being fed prompts generated by the Prompter agent and return a boolean on whether the reponse contains sensitive information or not.

## <a id="reporter"></a> Test Reporting [↑](#top)
### Static Catalogue Report
To generate, run:
```
python tools\generate_catalogue_report.py
```
A catalogue report can be generated under `/test-artifacts/catalogue-report.html` to view all the test cases that are defined in `/tests/metadata/catalogue.yaml`. This `catalogue.yaml` file will be serving as the source-of-truth for the all the test cases, containing all the metadata for each test case and their respective markers. Every pytest that is defined under the `/tests/` directory should ideally have a marker `@pytest.mark.tc_id("TC-ID")` defined to create a linkage with the YAML test cases.

### Test Execution Report
At the beginning of each test run, for every pytest that has a TC_ID marker defined, the modified PyTest framework will dynamically assign the rest of the markers from `catalogue.yaml` like test-level and component(s) onto the respective pytest. This will then trigger the specific set of tests accordingly and auto-generate a human-readable HTML report `/test-artifacts/report.html` after each test run.


### Test Artifacts
Screenshots will also be automatically captured at the point of failure now and stored in `test-artifacts/screenshots/` to help debug. The FastAPI backend and HTTP webserver log files will also be included after the CI as test artifacts.

## <a id="conversation-runner"></a> Conversation Runner [↑](#top)
This component orchestrates a conversation between the Prompter-LLM-Evaluator agents based on test flow inputs and logs the conversation into a turn-based format for the ease of human review upon failures.

## <a id="ci"></a> CI Pipeline [↑](#top)
The tests will be triggered to run automatically now following every push/PR into main branch and test artifacts are now available with each CI. Refer to `\.github\workflows\ci.yaml` for the configuration.
